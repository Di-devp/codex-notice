# 项目记忆

## 2026-06-11 Notice 当前关键架构记忆

### 技术栈
- 桌面应用：Tauri 2 + Rust + Tokio。
- 前端：Vue 3 + TypeScript + Vite + Pinia + Naive UI。
- 本地 HTTP：Axum，监听 `127.0.0.1:3746`。
- 存储：SQLite，默认路径为 `~/Library/Application Support/Notice/notice.db`。
- 敏感配置：飞书 webhook 与签名密钥走 macOS Keychain；SQLite 中只保存非敏感配置和脱敏展示信息。

### Hook 事件约定
Notice 管理的 Codex hooks 必须包含：

- `SessionStart`
- `UserPromptSubmit`
- `PermissionRequest`
- `PreToolUse`
- `PostToolUse`
- `Stop`

`UserPromptSubmit` 是红绿灯进入“运行中”的关键事件。缺少该事件时，用户新发消息不会被识别为新一轮任务开始。

### 红绿灯状态优先级
状态计算入口：`src-tauri/src/storage/mod.rs` 的 `traffic_widget_status()`。

优先级如下：

1. 红灯：最近 30 分钟内任一会话最新有效状态为失败。
2. 黄灯：无失败时，存在 pending approval 或用户确认状态。
3. 跑马灯：无失败、无待审批时，存在未结束的活动会话。
4. 绿灯：没有失败、没有待审批、没有运行中会话。

失败会话收到新的 `TASK_START` 后，会重新进入运行态，红灯消失。

### 飞书通知策略
飞书只通知两类事件：

- 需要用户审批。
- 一个会话中的任务成功结束。

工具调用过程中的普通成功事件不通知，失败或未完成的 Stop 不发送成功通知。

### 常用验证命令
- 前端构建：`pnpm build`
- Rust 测试：`cargo test --manifest-path src-tauri/Cargo.toml`
- Debug 打包：`pnpm tauri build --debug`
- 健康检查：`curl -sS http://127.0.0.1:3746/health`

### 用户当前需要注意
如果 `~/.codex/config.toml` 中搜索不到 `UserPromptSubmit`，需要在 Notice 的 Providers 页面重新安装 Codex Hooks，并新开 Codex 会话或重启 Codex 客户端测试。

## 2026-06-18 Notice-Mochi 集成记忆

### 协议
- Notice 向 Mochi 发送状态使用 HTTP GET：`GET {base_url}/pet/status?state=<state>&quota=<primary>&quota2=<secondary>`。
- `state` 取值来自红绿灯状态映射：
  - `red` -> `failed`
  - `yellow` 或存在待审批 -> `waiting`
  - `running` -> `running`
  - `green` 且存在完成事件 -> `complete`
  - 其他 `green` -> `ready`
- `quota` 是 primary 窗口格式化结果，例如 `5h 88%`。
- `quota2` 是 secondary 窗口格式化结果，例如 `7d 41%`。

### 状态存储
- `pet_enabled`：是否启用 Mochi 同步。
- `pet_base_url`：Mochi 基础地址，保存时会补 `http://` 并移除尾部 `/`。
- `pet_last_status`：最近一次桌宠同步状态文案。
- `pet_last_payload`：最近一次实际发送给 Mochi 的 `state|quota|quota2`，用于去重。
- `codex_usage_current`：Notice DB 中当前接受的 Codex 余量快照。
- `traffic_widget_manual_override`：拍照状态覆盖，支持 `ready`、`running`、`waiting`、`failed`、`complete`，空值表示真实状态。

### 触发策略
- Notice UI 可 800ms/4s 轮询 `get_traffic_widget_status()`，但不会每次都推 Mochi。
- `get_traffic_widget_status()` / `refresh_runtime_status()` 只有检测到 `codex_usage_changed` 时才同步一次 Mochi。
- Codex hook 事件仍可通过 `pet::sync_current_status()` 同步当前状态，但 `pet_last_payload` 会阻止完全相同的重复请求。

### Mochi 网络
- Mochi 固件运行 `WIFI_AP_STA` 双模式。
- `192.168.4.1` 是 Mochi 自己热点 `ClaWD-Mochi` 的 AP 地址。
- `WiFi.localIP()` 显示的地址才是连接家庭 WiFi 后的局域网地址，Notice 设置中应填写这个地址，例如 `http://192.168.0.102`。
- WiFi 凭据保存在 ESP32 Preferences/NVS；普通上传固件不会清掉，只有擦除整个 flash 才会丢配置。
