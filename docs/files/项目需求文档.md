# 项目需求文档

## 2026-06-11 红绿灯小组件与通知需求

### 用户目标
用户希望 Notice 不关注 Codex 每一次工具调用细节，而是关注“当前这一轮对话任务”的整体状态：是否开始执行、是否需要用户确认、是否失败、是否完成。

### 状态需求
- 当用户在 Codex 中发出新消息并触发任务执行时，红绿灯进入跑马灯。
- 如果任一运行中会话需要用户审批，显示黄灯。
- 如果任一会话失败，显示红灯。
- 当所有会话都成功结束且无待审批时，显示绿灯。
- 如果一个失败会话收到新的任务开始事件，说明用户已重新发起处理，红灯应清除并进入运行态。

### 交互需求
- 红绿灯小组件应无窗口边框。
- 红绿灯小组件应可直接拖动，不需要先点击激活。
- 小组件右键菜单支持置顶与隐藏。
- 右键菜单关闭行为应符合 macOS 用户预期。

### 通知需求
- 飞书只在需要用户审批时通知。
- 飞书只在一个会话任务成功结束后通知。
- 中间工具调用成功不通知，避免消息噪声。

## 2026-06-16 Codex 剩余用量展示需求

### 用户目标
用户希望 Notice 的红绿灯小组件不只显示 Codex 会话运行状态，还能直接看到当前 Codex 用户剩余用量，减少切回 Codex 查看状态的成本。

### 功能需求
- Notice 应在小组件中显示最近一次可获取的 Codex 用量窗口。
- 同时显示短周期窗口与长周期窗口，例如 5 小时窗口和 7 天窗口。
- 用量显示应表达“剩余额度百分比”，而不是已用百分比。
- 当没有可用 `rate_limits` 数据时，小组件应保持原有状态展示，不报错、不阻断红绿灯状态。

### 约束与接受标准
- 用量采集不得依赖读取或展示 Codex auth token。
- 用量展示不得改变红绿灯状态优先级。
- 小组件必须完整显示，右侧不得被裁切；新增内容后根容器、组件 CSS、Tauri 窗口尺寸必须同步。
- 旧运行实例可能保留旧窗口大小，因此显示小组件时应主动应用最新尺寸。

## 2026-06-18 多平台代码同步需求

### 用户目标
用户希望当前 Notice 代码可以一次性同步到所有已关联 Git 平台，包含 GitHub、Gitee、GitCode 等远端，避免各平台代码状态不一致。

### 接受标准
- 上传前运行基础验证，至少包括 `pnpm build` 与 `cargo test --manifest-path src-tauri/Cargo.toml`。
- 当前 `main` 分支应推送到所有已配置远端。
- 如果某个平台因认证失败未推送成功，需要明确记录失败平台、错误原因和恢复步骤。

## 2026-06-18 桌宠状态与余量显示需求

### 用户目标
用户希望 Mochi 桌宠能跟随 Notice 红绿灯状态展示 Codex 当前状态，并在桌宠屏幕上同步显示 Codex 剩余用量；同时避免因重复同步导致屏幕闪烁。

### 功能需求
- Notice 设置页提供 Mochi 桌宠配置：
  - 开关桌宠同步。
  - 保存 Mochi HTTP 地址。
  - 发送测试状态。
- Notice 同步给 Mochi 的 payload 必须包含：
  - 任务状态：`ready`、`running`、`waiting`、`failed`、`complete`。
  - 5h 余量文案。
  - 7d 余量文案。
- Mochi 固件需要在状态屏展示两行余量。
- Notice 需要持久化当前余量，只有余量签名变化时才触发余量相关同步。
- Notice 需要保存上次发送给 Mochi 的 payload，相同时跳过网络请求。

### 拍照需求
- Notice 设置页需要提供拍照状态覆盖。
- 覆盖状态包括：关闭覆盖、就绪、运行中、等待确认、失败、已完成。
- 覆盖只影响展示，不写入事件表，不影响统计。

### 体验要求
- Mochi `RUNNING` 动画应尽量减少闪烁。
- 固件应使用局部刷新绘制动态区域，避免每帧全屏清屏。
- 用户不应因为普通固件上传而必须重新配置家庭 WiFi；WiFi 凭据应继续依赖 ESP32 Preferences/NVS 保留。
